# Exact production solver

This directory contains the planner's exact production optimizer and the adapter that turns an
optimal mathematical solution into the machine banks, belts, and flows shown in the UI.

The public `solve` export is asynchronous and points to `solveExact` in `exact-planner.ts`. The
older heuristic modules remain only for compatibility tests and helper functions; the planner does
not use them to compute or reconstruct exact results.

## Goals and guarantees

For a valid recipe graph and planner input, the solver:

1. Enforces every minimum target and requested excess floor.
2. Never consumes more scarce raw material than is available.
3. Enforces exact conservation for every manufactured item.
4. Uses only clocks that make at least one machine input or output rate a whole number.
5. Enforces the selected conveyor capacity on every machine bank input and output.
6. Proves the global lexicographic optimum before returning a feasible plan.
7. Revalidates the returned solution with independent `BigInt` rational arithmetic.

If the minimums cannot be produced, the result is `INFEASIBLE`. If all supplied raw material cannot
be consumed, the solver returns the plan with the greatest feasible consumption. Cancellation
returns `CANCELLED`; a merely feasible incumbent is never presented as globally optimal.


The main entry points are:

- `exact-planner.ts`: planner input normalization and UI result construction.
- `exact/optimizer.ts`: CP-SAT model construction and lexicographic solving.
- `exact/validation.ts`: independent post-solve verification.

## Exact arithmetic

All model rates are represented by `Rational` from `exact/rational.ts`:

```text
numerator: bigint
denominator: positive bigint
```

Values are normalized by their greatest common divisor. Decimal user inputs are parsed from their
decimal string representation, so values such as `0.1` become `1/10` rather than an approximation
produced by repeated binary floating-point calculations.

OR-Tools CP-SAT accepts integer coefficients. `exact/integer-linear.ts` converts each constraint
row and objective independently:

1. Compute the least common multiple of that row's rational denominators.
2. Multiply every coefficient by the local scale.
3. Verify that every scaled coefficient is integral.
4. Verify coefficients, variable bounds, and the largest possible expression value remain within
   JavaScript's exact safe-integer range.

Using a local scale avoids the very large least common multiple that a single model-wide scale
would create. Floating-point values are introduced only when the proven result is converted for UI
display.

## Recipe graph validation

`exact/recipe-graph.ts` validates the assumptions used to make the search finite and unambiguous:

- Item and recipe IDs are unique.
- Every recipe reference points to a known item.
- Recipe durations and item amounts are positive and finite.
- Every recipe has exactly one output.
- Every manufactured item has exactly one producer.
- Recipe dependencies form a directed acyclic graph.
- Every production chain reaches at least one scarce raw resource.

The returned graph contains producer indexes and a topological recipe order. Invalid data is
rejected before any optimization model is built.

The current data set intentionally has one standard recipe per product. Alternate recipes,
byproducts, and multiple-output recipes require a broader model and are rejected rather than being
silently interpreted incorrectly.

## Legal clock generation

Clocks are recipe-specific, exact fractions in the interval:

```text
0 < clock <= 1
```

Overclocking is not considered. For a recipe port with a full-clock rate `r` items/min, every legal
clock contributed by that port has the form:

```text
clock = k / r
where k is an integer and 1 <= k <= floor(r)
```

`exact/underclocks.ts` unions the clocks generated by every input and output port, reduces duplicate
fractions, and sorts them. This is the complete finite set for which at least one port has an
integer items/min rate.

For Quickwire, one constructor at full clock consumes `12` Caterium Ingots/min and produces `60`
Quickwire/min. The clock `5/6` is legal because it gives:

```text
input  = 12 * 5/6 = 10 Caterium Ingots/min
output = 60 * 5/6 = 50 Quickwire/min
```

Four constructors at `5/6` therefore consume `40/min` and produce exactly `200/min`.

## Finite machine bounds

`exact/bounds.ts` expands each recipe cycle through the validated dependency graph to obtain its
transitive scarce-raw requirement. For every required scarce raw `j`, the available amount gives
an upper bound:

```text
maximum cycles <= available[j] / raw required per cycle[j]
```

The tightest raw bound limits recipe activity. Dividing by the recipe's full-clock cycles/min gives
the maximum effective machine count. Dividing again by the recipe's smallest legal clock gives a
finite physical-machine bound.

Recipes that have no scarce-resource dependency are rejected because unconstrained excess
production would otherwise make the search domain unbounded.

## Machine-bank patterns

`exact/bank-patterns.ts` enumerates every relevant equal-clock bank:

```text
(recipe, legal clock, machine count)
```

A bank is legal when:

- Its machine count is `2^a * 3^b`, including one machine. These counts can be fed by nested
  two-way and three-way splitters.
- Every input and output belt rate is at most the selected conveyor capacity.
- Its effective activity stays within the raw-derived recipe bound.

There is no arbitrary splitter-depth cutoff. Raw and belt bounds make the set finite.

Each pattern stores exact per-bank input rates, output rates, effective machines, and cycles/min.
The optimizer's integer variable is the number of copies of that bank pattern.

### Safe dominance reductions

The full legal domain contains many equivalent representations. The optimizer removes only
representations that cannot occur in a lexicographically optimal solution:

- If two single banks have identical effective production, keep the one with fewer physical
  machines.
- If `k` copies of a smaller bank exactly equal an available larger bank using no more machines,
  at most `k - 1` interchangeable copies of the smaller bank are needed. Any additional copies can
  be replaced by the larger bank with equal flow and no worse earlier objective.

These reductions remove symmetry; they do not remove a non-dominated production or layout.

## CP-SAT model

The model uses two related representations to avoid forcing CP-SAT to reason about thousands of
equivalent bank partitions during the first two objective passes.

### Compact production variables

Production variables represent recipe activity with a small set of one-machine rate generators.
A generator is omitted when its effective rate is an integer multiple of a smaller generator.

These variables drive:

- Scarce-raw capacity constraints.
- Exact manufactured-item conservation.
- Target and excess withdrawals.
- The first two optimization objectives.

This compact representation chooses what is produced without yet choosing how identical activity
is partitioned into physical banks.

### Physical bank variables

Bank variables represent the complete non-dominated bank-pattern domain. After raw utilization and
weighted target output are fixed at their proven optima, the model adds an exact equality for each
recipe:

```text
sum(production generator effective rates)
  = sum(selected bank effective rates)
```

The remaining objective passes then choose the globally best physical representation of the
already-optimal production rates.

## Conservation and withdrawals

For every manufactured item `i`, the model adds:

```text
sum(recipe outputs of i)
  = sum(recipe inputs of i)
  + target withdrawal of i
  + excess withdrawal of i
```

This equality is the main protection against invented production, disappearing intermediates, and
incorrectly reconstructed inputs.

Additional rules:

- A target withdrawal exists only for requested target items and is bounded below by its minimum.
- Every manufactured non-ingot has an excess/storage withdrawal variable. A requested excess rate
  is its lower bound.
- Ingots have no excess variable, so raw utilization cannot be increased merely by storing unused
  ingots.
- Scarce raw consumption is bounded by availability.
- Unlimited raw items are permitted as unconstrained external sources.

Target and excess are separate withdrawals even when they refer to the same item.

## Routing complexity

The bank model treats each selected bank as one output lane. Boolean activity variables identify
which recipe consumers, targets, and excess sinks are active for each item.

For an item with `D` active destinations and `L` selected output lanes, the minimum number of
additional three-output splitter devices is:

```text
max(0, ceil((D - L) / 2))
```

Each three-output splitter replaces one lane with three, increasing the number of available
branches by two. CP-SAT models this relation with an integer routing-device variable and exact
linear constraints.

Internal bank complexity is also counted. For a `2^a * 3^b` machine bank,
`equalLaneTreeDevices()` counts the splitter tree used by each input and the corresponding merger
tree used by the output. A recipe with `n` inputs therefore contributes:

```text
equal-lane tree devices * (n + 1)
```

The final complexity objective minimizes internal splitters, internal mergers, and downstream
routing splitters together.

## Lexicographic objective

The objectives are solved in strict priority order:

1. **Maximize scarce raw use**  
   Maximize the sum of consumed scarce items/min.
2. **Maximize weighted target output**  
   Maximize `sum(target rate * target weight)`.
3. **Minimize physical machines**
4. **Minimize machine groups**
5. **Minimize splitter and merger devices**
6. **Stable deterministic tie-break**  
   Prefer a stable bank-pattern order so repeated solves return the same equivalent optimum.

This is not a weighted blend. After each pass returns `OPTIMAL`, its exact objective value is added
to the model as an equality before the next pass begins. A lower-priority improvement can therefore
never trade away any amount of a higher-priority objective.

There is no default time limit. The solver continues until CP-SAT proves `OPTIMAL` or
`INFEASIBLE`, unless the caller cancels it.

## Search-performance measures

Exact optimization is combinatorial, but several completeness-preserving measures keep normal
browser inputs practical:

- Compact production generators isolate flow selection from bank partitioning.
- Dominated bank patterns and interchangeable multiplicities are removed.
- Physical bank domains are tightened after the machine-count optimum is known.
- Each solved phase supplies a complete or canonical solution hint to the next phase.
- CP-SAT uses all but one reported logical core, capped at eight workers; callers can override the
  worker count for benchmarks and constrained devices.
- The fixed CP-SAT random seed and stable final objective keep equivalent output repeatable across
  parallel solves.

These measures change search order and remove symmetry only; they do not turn the optimizer into a
hill climber or impose an iteration cap.

## Independent validation

`exact/validation.ts` does not trust the CP-SAT row scaling or reported totals. It recomputes the
solution from recipe data and selected banks using `Rational` arithmetic and verifies:

- Every bank clock is legal.
- Every machine count is splitter-compatible.
- Every bank input and output fits the selected belt.
- Reported bank rates match recipe arithmetic.
- Selected activity stays within raw-derived bounds.
- Every target minimum and excess floor is met.
- Ingots have zero excess.
- Every manufactured item is exactly conserved.
- Raw use and leftover values are correct.
- All reported objective values match recomputed values.

The planner receives a feasible result only if CP-SAT reports `OPTIMAL` and this independent
validator returns no issues.

## Result construction

`exact-planner.ts` converts the validated exact solution into `SolveResult`:

1. Expand each selected bank multiplicity into visible machine groups.
2. Use the selected clock and rates directly; do not quantize or repack them.
3. Create one belt-capped input and output lane per selected bank.
4. Collect exact recipe, target, and excess destinations.
5. Allocate every output lane completely across those destinations.
6. Mark multi-destination lanes as demand-balanced manifolds, where downstream backpressure holds
   the solved rates.
7. Build raw edges, stage edges, recipe usage, item totals, and production-chain ordering.

Lane allocation throws if any selected production is unallocated or any destination is short. The
display network is therefore a projection of the same optimized variables, not a second solver.

## Browser execution and cancellation

`or-tools-wasm` runs CP-SAT through its browser Worker bridge. The React UI awaits `solveExact`
without blocking rendering and uses an `AbortSignal` for cancellation. `cancelExactSolve()` forwards
the request to the active CP-SAT worker. The six objective phases emit progress events; the planner
shows the active phase and elapsed time when a solve lasts longer than 15 seconds.

The threaded WebAssembly runtime requires:

```text
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
```

`vercel.json` configures these headers for Vercel. Other static hosts must provide equivalent
headers. The pinned package patch in `patches/or-tools-wasm@0.9.1.patch` resolves emitted runtime
asset paths correctly in a Next.js static export.

## Extending the solver

When adding an item or recipe:

1. Add the data to `src/data`.
2. Keep recipe amounts and durations representable as exact decimal values.
3. Confirm the graph still has a unique producer per manufactured item and no cycles.
4. Add clock/rate tests for unusual fractional ports.
5. Add a conservation regression for any new multi-input chain.
6. Run the exact-domain tests, full checks, and static build.

Do not add a clock to a global allowlist or manually quantize a rate. Legal clocks and bank patterns
must continue to be derived from recipe arithmetic so completeness and conservation remain aligned.

## Current scope

The exact guarantees apply within these explicit modeling assumptions:

- Standard clocks are underclocks only: `(0, 100%]`.
- One producer and one output per manufactured item.
- Acyclic recipes with a finite scarce-resource dependency.
- Two-way and three-way equal-split machine banks.
- Conveyor capacity applies independently to every bank input and output.
- Raw-use value is the sum of scarce items/min; different raw item types are not assigned economic
  conversion values.
- Solver coefficients and bounds must fit JavaScript's exact safe-integer range after local scaling.

Worst-case runtime remains exponential because machine grouping and routing contain integer
decisions. The solver reports global optimality only when CP-SAT has completed its proof.
